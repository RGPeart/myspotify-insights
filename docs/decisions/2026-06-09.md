# 2026-06-09

## 1. Schema registry lives in `src/schemas/`, separate from `src/models/`

**Status:** Accepted

**Context:**
Feature 8 (Schema Registry & Schema Evolution) introduces Pydantic models as the
canonical contract for each pipeline dataset. The PRD's example code places these
in `src/models/` and imports `src.models.silver`. However, `src/models/` already
holds the ML recommendation code (`train.py`, `predict.py`). Putting data-layer
schema contracts in the same package would conflate two unrelated concerns —
"what shape is this dataset" vs. "how is the recommender trained".

**Decision:**
Create a dedicated `src/schemas/` package instead of following the PRD literally:

- `silver.py` / `gold.py` — Pydantic contracts for the silver and gold datasets.
- `registry.py` — `SchemaSpec` dataclass + `SCHEMA_REGISTRY` mapping `layer/name`
  to a spec, plus `build_json_schema()` which is the single code path used by both
  the generation script and the drift test.
- `validate.py` — `validate_dataframe()` and `SchemaValidationError`.

**Consequences:**
- Imports are `src.schemas.silver` etc., diverging from the PRD's `src.models.silver`.
  The separation is worth the divergence; Feature 9 (Data Contracts) will import from
  `src/schemas/` rather than `src/models/`.
- `src/models/` remains ML-only; no risk of a data-contract change touching model code.

## 2. Schema coverage is silver + gold; bronze is excluded

**Status:** Accepted

**Context:**
The PRD's illustrative schemas (e.g. a `SilverTrack` that merges audio features into
the track row, and a `fact_listening_history` table) do not match the real pipeline.
The actual datasets are three silver Parquet files (`tracks`, `audio_features`,
`artists`) and three gold tables (`dim_tracks`, `dim_artists`, `fact_audio_features`).
Bronze is raw, nested Spotify API JSON whose shape Spotify controls.

**Decision:**
Model the **six real silver and gold datasets** exactly as produced by
`src/etl/bronze_to_silver.py` and `dbt/models/gold/*.sql`. Required fields and ranges
mirror the existing data-quality checks and dbt schema tests so the Python contract
and the dbt tests agree on validity. Bronze is intentionally not modeled.

**Consequences:**
- The contracts reflect reality, not the PRD's illustrative examples. A future
  bronze contract would need a deliberately loose model and is out of scope here.
- Runtime validation is wired only at the silver boundary (in `bronze_to_silver.run()`),
  the one place the Python pipeline materializes a DataFrame it owns. Gold validity is
  enforced by the existing dbt tests; the gold Pydantic models exist for documentation,
  JSON Schema generation, and reuse by downstream consumers.

## 3. JSON Schema drift is caught by a pytest check, not a pre-commit hook

**Status:** Accepted

**Context:**
The PRD calls for "a pre-commit hook or CI check that fails if Pydantic models and
JSON Schema files are out of sync." The repo's CI already runs `pytest`, and there is
no existing pre-commit infrastructure.

**Decision:**
Implement the sync guard as a parametrized test in `tests/test_schemas.py` that
regenerates each schema via `build_json_schema()` and asserts equality with the
committed file. JSON Schema files are generated by `python scripts/generate_schemas.py`.

**Consequences:**
- No new tooling (pre-commit) to install or document; the guard runs wherever pytest
  runs. The trade-off is that drift is caught at test time rather than commit time —
  acceptable since CI runs the suite on every PR.
- Contributors who change a model must regenerate and commit the JSON Schema, or the
  suite fails with a message pointing at `scripts/generate_schemas.py`.
