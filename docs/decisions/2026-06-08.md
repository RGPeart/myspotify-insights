# 2026-06-08

## 1. Switch ingestion source from genre search to personal listening data

**Status:** Accepted

**Context:**
The previous ingestion in `src/ingestion/spotify_client.py` authenticated with
Client Credentials and discovered tracks via a fixed 8-genre search list. The
resulting catalog had no relationship to any real user's taste, so the
recommendation model in `src/models/train.py` was forced to rely on synthetic
user profiles. The project goal is a personalised recommendation system, so
the ingestion source should be the authenticated user's actual listening
history.

**Decision:**
Move to Spotify's **Authorization Code** (user OAuth) flow and add personal
endpoints as the primary ingestion source:

- `/me` → bronze `user_profile/`
- `/me/top/tracks` per time_range → bronze `user_top_tracks/{time_range}/`
- `/me/top/artists` per time_range → bronze `user_top_artists/{time_range}/`
- `/me/following` (artists) → bronze `followed_artists/`

Time ranges: **`short_term` (~4 weeks) + `medium_term` (~6 months)** only.
`long_term` is skipped — for a personalised recommender, recent and current
taste are the load-bearing signal; lifetime averages drown that out.

Genre search is retained as **catalog expansion**: the genres of the user's
top + followed artists are aggregated, ranked by frequency, and the top
`derived_genres_max` (default 10) drive `sp.search(q="genre:X")` instead of
the hardcoded list. The hardcoded list is renamed `fallback_genres` and used
only when no personal genres are derivable (new account, scope revoked, etc.).

Top tracks are also merged into the standard `tracks/` data type (deduped via
the existing manifest) so they reach silver/gold even if the catalog search
misses them. Top + followed artists are similarly merged into `artists/`,
avoiding redundant `/artists` calls for IDs we already have full payloads for.

**Consequences:**
- Bronze layer gains four new directory types. No silver/dbt/gold schema
  changes — the standard `tracks/`, `audio_features/`, `artists/` files keep
  the same shape because `/me/top/tracks` and `/me/top/artists` return the
  same Track / Artist objects that the existing `transform_*` functions
  already read.
- The new directories (`user_profile/`, `user_top_*/{time_range}/`,
  `followed_artists/`) are **not consumed by silver yet** — landed only for
  the deferred ML follow-up.
- ML model integration (a `fact_user_top_tracks` gold table and the rewrite
  of `train.py` to use it instead of synthetic profiles) is explicitly
  deferred. This decision changes the *source* of catalog data; the model
  rewrite is a separate, larger change.

## 2. Headless OAuth via one-time local login + refresh token in `.env`

**Status:** Accepted

**Context:**
Authorization Code flow requires a browser redirect, but Airflow workers run
headless in Docker. The refresh token only needs to be obtained once and then
can be exchanged for short-lived access tokens indefinitely (until the user
revokes the app).

We considered three options:

1. One-time local login → refresh token in `.env` (chosen).
2. Persistent token cache file in `data/spotify/.cache` (volume-mounted).
3. Airflow Connection / Variable storing the refresh token.

**Decision:**
A one-time CLI (`python -m src.ingestion.spotify_auth_login`) runs the browser
flow locally, prints the refresh token to stdout, and instructs the developer
to paste it into `.env` as `SPOTIFY_REFRESH_TOKEN`. At ingest time,
`src/ingestion/spotify_auth.get_authenticated_client()` reads that env var,
seeds spotipy's `MemoryCacheHandler` with it, and returns a `spotipy.Spotify`
that auto-refreshes the access token on first API call.

Nothing token-related is written to disk inside the container; refresh happens
entirely in memory. The refresh token itself lives only in `.env` (which is
gitignored — `.env.example` is the tracked template).

Failure modes:
- Missing `SPOTIFY_REFRESH_TOKEN` → `RuntimeError` on
  `SpotifyIngestionClient.__init__` with an actionable message pointing to
  the login script.
- Revoked / invalid refresh token → spotipy's auth error propagates up; user
  must re-run the login script.

**Consequences:**
- No Docker volume changes; no Airflow Connection setup; no token-cache file
  to leak credentials onto disk. The trade-off is that revoking and rotating
  the refresh token is a manual step (re-run the login script, update `.env`,
  restart Airflow).
- Tests mock `get_authenticated_client` directly; no real OAuth is exercised
  in CI.
- The login script is itself a development tool, not a runtime dependency —
  it's only imported when invoked.
