# Decisions — 2026-05-18

## 1. Drop bronze rows with null audio features at the silver gate

**Status:** Accepted

**Context:** The `silver/audio_features` quality gate requires `danceability`, `energy`, and `tempo` to be non-null on every row. Spotify's `/audio-features` endpoint occasionally returns null blobs for podcasts, local files, and relinked/regional tracks, which caused `_bronze_to_silver` to fail with `DataQualityError`.

**Decision:** In `src/etl/bronze_to_silver.py`, `transform_audio_features` now drops rows where any of `danceability`, `energy`, or `tempo` is null and emits a warning log with the dropped row count. Filtering happens before deduplication.

**Consequences:** The pipeline is no longer brittle to Spotify's null-feature edge cases. Dropped tracks become invisible to the recommender, which is acceptable — they are unusable without features. The warning log is the monitoring signal; a sudden spike in drops indicates an upstream ingestion regression worth investigating.

## 2. Remap Marquez to port 5002 (host and internal)

**Status:** Accepted

**Context:** macOS AirPlay Receiver (`ControlCenter`) binds `*:5000` by default since Monterey, blocking Marquez's upstream default port. Disabling AirPlay was rejected because it modifies user system settings and would silently break again on macOS updates.

**Decision:** Marquez listens on 5002 (admin on 5003) inside the container, with matching host-side mapping `5002:5002`. `OPENLINEAGE_URL`, the Marquez healthcheck, all docs, and the dashboard's `MARQUEZ_URL` default all reference 5002.

**Consequences:** Marquez deviates from upstream documentation defaults — anyone consulting external Marquez docs must mentally remap port references. The Spotify OAuth callback URI (`localhost:5000/callback` in `.env`) remains a latent AirPlay conflict; it has not been exercised because existing refresh tokens bypass the redirect flow. A future ADR should address the OAuth port when a fresh auth flow is required.

## 3. Always start the stack with `--profile lineage`

**Status:** Accepted

**Context:** Marquez services are gated behind `profiles: [lineage]` in `docker-compose.yaml`. Running `docker compose up -d` (no profile) followed later by `docker compose --profile lineage up -d` produced DNS resolution failures for the `marquez` hostname from inside Airflow containers, because Airflow's networking initialized before Marquez existed in the compose network.

**Decision:** Use `docker compose --profile lineage up -d` and `docker compose --profile lineage down` as the canonical start and stop commands. Mixing profile-on and profile-off invocations within a single session is not supported.

**Consequences:** Marquez always starts alongside Airflow, consuming approximately 1 GB of RAM under amd64 emulation on Apple Silicon. This is an acceptable trade-off compared to debugging recurring network-isolation bugs. If lineage tracking is later disabled by default, remove the profile gate rather than reintroducing two-step startup.

## 4. Accept dataset-lineage gap for current TaskFlow DAG

**Status:** Accepted (interim)

**Context:** The OpenLineage Airflow provider auto-extracts dataset lineage only for operators with built-in extractors (Postgres, BigQuery, S3, etc.). `dags/spotify_etl_dag.py` uses TaskFlow `@task` decorators calling internal Python modules that read and write local Parquet, for which no extractor exists. Marquez therefore records job runs and durations but shows an empty dataset graph.

**Decision:** Accept this gap for now. Job-run observability is sufficient for the current Feature 6 deliverable. Adding dataset-level lineage is deferred until either (a) a follow-up feature explicitly scopes it, or (b) the ETL is refactored onto operators with native extractors.

**Consequences:** Marquez UI is useful for run history and failure correlation but not for end-to-end data lineage visualization. When dataset lineage is later prioritized, the most direct path is adding explicit `inputs=` / `outputs=` declarations on each `@task` or wiring custom extractors against the Parquet reads/writes in `src/utils/parquet_io.py`.
