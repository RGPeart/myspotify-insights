# 2026-05-19

## 1. Replace Silver → Gold pandas transform with dbt + DuckDB

**Status:** Accepted

**Context:**
The existing `src/etl/silver_to_gold.py` builds `dim_tracks`, `dim_artists`, and
`fact_audio_features` in pandas. Dimensional modeling (joins, popularity
aggregation, genre fill) is fundamentally a relational workload — SQL is a
better fit than pandas. dbt + DuckDB also brings model-level tests, generated
docs, and an interactive lineage DAG that complements the OpenLineage/Marquez
job-level lineage from Feature 6.

**Decision:**
Introduce a dbt project at `/dbt/` using the `dbt-duckdb` adapter. Gold models
are materialised as external Parquet files via `materialized='external'` so the
existing API and dashboard (which read from `data/gold/*.parquet`) continue to
work unchanged. The Airflow DAG calls `dbt deps`, `dbt run`, `dbt test` via
`BashOperator` after `_bronze_to_silver`, replacing the `_silver_to_gold` task.

**Consequences:**
- `src/etl/silver_to_gold.py` is no longer invoked from the DAG. It is left in
  the repo for reference but is dead code; a follow-up should remove it once
  dbt has been run end-to-end against real data.
- `data/spotify.duckdb` becomes a build artefact during `dbt run`. It is added
  to `.gitignore`.
- New contributors need to run `cd dbt && dbt deps` once to install `dbt_utils`
  before `dbt run`. The DAG handles this automatically via the `dbt_deps` task.

## 2. PRD Feature 7 spec divergence — match actual silver schema

**Status:** Accepted

**Context:**
The PRD spec for Feature 7 references `silver/listening_history/*.parquet`,
`track_popularity`/`artist_popularity` columns in silver, and a
`fact_listening_history` gold table. None of these exist in the actual
pipeline. The real silver tables are `tracks`, `artists`, `audio_features`
(single Parquet files, not partitioned), with a `popularity` column in both
tracks and artists. The existing gold layer is `dim_tracks`, `dim_artists`,
`fact_audio_features`.

**Decision:**
Implement dbt models that reproduce the existing pandas semantics rather than
the PRD's hypothetical `fact_listening_history`:

- `stg_silver_tracks` renames `popularity` → `track_popularity`.
- `stg_silver_artists` renames `popularity` → `artist_popularity` and
  `name` → `artist_name`.
- `stg_silver_audio_features` passes through.
- `dim_tracks` computes composite popularity as
  `0.6 × track_popularity/100 + 0.4 × artist_popularity/100`, with
  artist_popularity median-imputed for unmatched rows (matching the pandas
  fallback in `silver_to_gold.py`).
- `dim_artists`, `fact_audio_features` replicate the existing column layout.

**Consequences:**
- The API and Streamlit dashboard keep reading the same Parquet files with
  the same schema — no downstream changes required.
- The PRD remains useful as the design rationale but should not be read as the
  canonical table list. The dbt project is the source of truth for gold layer
  structure going forward.
- Adding a real `fact_listening_history` (sourced from Spotify recently-played
  endpoint) is a future ingestion-side enhancement, not part of this feature.
